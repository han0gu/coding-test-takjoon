# Codex 피드백

## 문제 접근

이번 피드백의 핵심은 문제 전체 풀이가 아니라, **최댓값 초기값을 어떻게 정할 것인가**입니다.
길이가 `K`인 연속 구간의 합을 비교하는 문제이므로, 비교 기준도 실제로 가능한 `K`개 구간의 합이어야 합니다.

예를 들어 다음 입력을 생각해봅시다.

```text
5 3
3 -2 -4 -9 0
```

`K = 3`이므로 확인해야 하는 구간은 다음 3개입니다.

| 시작 인덱스 | 구간 | 합 |
| --- | --- | --- |
| `0` | `3 + (-2) + (-4)` | `-3` |
| `1` | `-2 + (-4) + (-9)` | `-15` |
| `2` | `-4 + (-9) + 0` | `-13` |

이 중 가장 큰 값은 `-3`입니다.

## 개선한 점

### 1. 최댓값 초기값을 하드코딩하지 않기

처음에는 최댓값의 초기값을 아래처럼 임의의 값으로 두었습니다.

```python
max = -101
```

하지만 연속 구간의 합은 온도 하나보다 훨씬 작아질 수 있습니다.

예를 들어 다음 입력을 생각해봅시다.

```text
3 3
-100 -100 -100
```

가능한 구간은 하나뿐이고, 정답은 `-300`입니다.
그런데 `max = -101`로 시작하면 `-300`보다 `-101`이 더 크기 때문에 최댓값이 갱신되지 않습니다.
그래서 잘못된 답인 `-101`이 출력될 수 있습니다.

그렇다면 `-10000`이나 `-10001`처럼 더 작은 값을 쓰면 될까요?
이 문제의 조건만 보면 온도는 최소 `-100`, 날짜 수는 최대 `100`이므로 가능한 가장 작은 구간 합은 `-100 * 100 = -10000`입니다.

따라서 아래처럼 작성해도 이 문제에서는 동작할 수 있습니다.

```python
max = -10001
```

하지만 이 방식은 문제 조건을 코드에 직접 박아 넣는 방식입니다.
나중에 조건을 잘못 기억하거나, 비슷한 문제에서 범위가 바뀌면 같은 실수를 반복할 수 있습니다.

이번에는 첫 번째 구간의 합을 최댓값의 초기값으로 사용하기로 했습니다.

```python
max = sum(temperatures[:k])
```

이렇게 하면 최솟값 범위를 따로 계산하지 않아도 되고, 실제로 가능한 값에서 비교를 시작할 수 있습니다.
즉, `-10001`보다 더 문제의 구조에 맞는 초기값입니다.

### 2. 첫 번째 구간을 중복 계산하지 않기

또한 첫 번째 구간 합을 이미 구했기 때문에, 반복문은 두 번째 구간부터 확인하도록 바꾸었습니다.
이전보다 중복 계산이 줄어들고 코드의 의도도 더 분명해졌습니다.

```python
for i in range(1, n-k+1):
    temp_sum = sum(temperatures[i:i+k])
```

예를 들어 `n = 5`, `k = 3`이면 가능한 시작 인덱스는 `0`, `1`, `2`입니다.

- `0`번 구간은 이미 `max = sum(temperatures[:k])`에서 계산했습니다.
- 반복문에서는 `1`, `2`번 구간만 추가로 확인하면 됩니다.
- 그래서 `range(1, n-k+1)`은 `range(1, 3)`이 되고, 실제로 `1`, `2`를 돕니다.

만약 `n == k`라면 가능한 구간은 전체 배열 하나뿐입니다.
이때는 `range(1, 1)`이 되어 반복문이 실행되지 않고, 처음 구한 구간 합이 그대로 정답이 됩니다.

## 현재 풀이의 핵심

현재 풀이 흐름은 다음과 같습니다.

```python
max = sum(temperatures[:k])

for i in range(1, n-k+1):
    temp_sum = sum(temperatures[i:i+k])
    if max < temp_sum:
        max = temp_sum
```

정리하면 다음 순서입니다.

1. 첫 번째 길이 `K` 구간의 합을 구한다.
2. 그 값을 현재까지의 최댓값으로 둔다.
3. 두 번째 구간부터 마지막 구간까지 차례대로 합을 구한다.
4. 더 큰 합이 나오면 최댓값을 갱신한다.

## 현재 풀이가 최적인가?

현재 풀이는 각 시작 위치마다 `K`개를 다시 더합니다.
그래서 시간 복잡도는 `O(NK)`입니다.

이 문제는 `N`이 최대 `100`이라서 현재 풀이로도 충분히 통과할 수 있습니다.
즉, 지금 코드는 이 문제의 조건에서는 읽기 쉽고 실수하기 어려운 좋은 풀이입니다.

다만 더 효율적인 방법도 있습니다.
연속 구간이 한 칸 이동할 때는 왼쪽 값 하나가 빠지고 오른쪽 값 하나가 새로 들어옵니다.
이 성질을 이용하면 매번 `K`개를 다시 더하지 않고 구간 합을 갱신할 수 있습니다.

```python
current_sum = sum(temperatures[:k])
max_sum = current_sum

for i in range(k, n):
    current_sum = current_sum - temperatures[i-k] + temperatures[i]
    if max_sum < current_sum:
        max_sum = current_sum
```

이 방식은 슬라이딩 윈도우라고 부르고, 시간 복잡도는 `O(N)`입니다.
하지만 이 문제에서는 입력 크기가 작기 때문에, 처음에는 현재 풀이처럼 직접 구간 합을 계산하는 방식으로 풀어도 괜찮습니다.

## 다음 문제에 적용할 점

최댓값이나 최솟값을 구할 때는 임의의 큰 값이나 작은 값으로 시작하기보다, 실제 입력에서 만들 수 있는 첫 번째 값을 기준으로 잡는 습관을 들이면 실수를 줄일 수 있습니다.

특히 다음처럼 생각하면 좋습니다.

- "내가 비교하려는 값이 구간 합인가?"
- "초기값도 실제로 가능한 구간 합인가?"
- "이미 계산한 구간을 반복문에서 다시 계산하고 있지는 않은가?"
