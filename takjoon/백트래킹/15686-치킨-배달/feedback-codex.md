# 치킨 배달 피드백

## 문제 조건 요약

- 집과 치킨집의 좌표가 주어지고, 치킨집 중 최대 `M`개를 남긴다.
- 각 집의 치킨 거리는 남긴 치킨집 중 가장 가까운 치킨집까지의 거리다.
- 도시의 치킨 거리는 모든 집의 치킨 거리의 합이며, 이 값을 최소화해야 한다.
- 치킨집 개수는 최대 `13`개이고, 집의 개수는 최대 `2N`개다.

## 현재 풀이의 좋은 점

현재 풀이는 치킨집을 선택하거나 선택하지 않는 방식으로 상태를 탐색한다.

```text
현재 치킨집을 선택한다
현재 치킨집을 선택하지 않는다
```

치킨집 수가 최대 `13`개라서 이 방향의 완전탐색은 충분히 가능하다. 또한 단순히 선택한 치킨집 목록만 들고 다니는 것이 아니라, 각 집의 현재 최소 치킨거리를 상태로 들고 다닌 점이 좋다.

```python
chick_distances
```

새 치킨집을 선택할 때마다 모든 집에 대해 다음 값만 갱신한다.

```text
min(기존 최소 거리, 새 치킨집까지의 거리)
```

이렇게 하면 선택된 치킨집 조합 전체를 매번 다시 계산하지 않고, 이전 상태에서 계산한 집별 최소 거리를 재활용할 수 있다.

정확히는 "도시의 치킨 거리 총합" 하나를 재활용하는 것이 아니라, **각 집별 현재 최소 치킨거리 리스트**를 재활용하는 것이다. 총합만 들고 있으면 새 치킨집을 추가했을 때 어떤 집의 거리가 얼마나 줄어드는지 알 수 없기 때문이다.

## combinations 풀이와 비교

이 문제는 직관적으로 `combinations`가 먼저 떠오르기 좋은 문제다. 치킨집 중 `M`개를 고른 뒤, 각 조합마다 도시의 치킨 거리를 계산하면 된다.

`combinations` 풀이의 장점은 코드가 단순하고 의도가 명확하다는 점이다.

```text
M개 치킨집 조합 생성
각 조합에 대해 도시 치킨 거리 계산
최솟값 갱신
```

단점은 각 조합마다 거리 계산을 다시 해야 한다는 점이다.

```text
O(조합 수 * 집 개수 * 선택한 치킨집 수)
```

하지만 이 문제에서는 제한이 작다.

```text
치킨집 수 C <= 13
집 수 H <= 2N <= 100
```

조합 수가 가장 많은 경우도 대략 다음 정도다.

```text
13C6 = 1716
13C7 = 1716
```

따라서 단순 조합 풀이로도 충분히 통과 가능한 크기다. `itertools.combinations`는 조합을 리스트로 한 번에 모두 만드는 함수가 아니라 이터레이터이므로, `for`문으로 돌면 조합을 하나씩 생성한다. 메모리 부담도 크지 않다.

## 백트래킹 풀이의 장단점

현재처럼 백트래킹으로 풀면 거리 상태를 재활용할 수 있다는 장점이 있다. 새 치킨집 하나를 선택할 때마다 모든 집의 최소 거리만 갱신하면 되므로, 조합마다 처음부터 다시 계산하는 것보다 중간 계산을 활용할 수 있다.

다만 선택/미선택 백트래킹은 중간 상태를 많이 만든다.

예를 들어 치킨집 수가 `C = 13`, 선택 수가 `M = 6`이면 정확히 `6`개를 고르는 조합 수는 다음과 같다.

```text
13C6 = 1716
```

하지만 선택/미선택 방식은 `0`개 선택, `1`개 선택, `2`개 선택, ..., `6`개 선택 상태를 거쳐 간다.

```text
13C0 + 13C1 + 13C2 + ... + 13C6
= 4096
```

즉 백트래킹은 거리 계산을 재활용할 수 있지만, 조합 결과에 도달하기 전의 중간 상태도 처리한다. 그래서 항상 `combinations`보다 빠르다고 단정할 수는 없다.

성능은 다음 두 가지를 함께 봐야 한다.

```text
탐색하는 상태 수
상태 하나를 처리하는 비용
```

## M개를 선택한 경우만 봐도 되는 이유

문제에서는 치킨집을 "최대 `M`개" 고른다고 되어 있다. 하지만 최적해는 항상 정확히 `M`개를 고른 경우 중 하나에 있다.

어떤 치킨집 집합을 `S`라고 하자. 여기에 새 치킨집 하나를 추가해서 `S'`를 만들면, 각 집의 치킨 거리는 다음처럼 바뀐다.

```text
min(기존 최단거리, 새 치킨집까지의 거리)
```

즉 각 집의 거리는 줄어들거나 그대로 유지된다. 늘어날 수는 없다. 따라서 도시의 치킨 거리 합도 줄어들거나 그대로 유지된다.

그러므로 `M`개보다 적게 선택한 상태가 좋아 보이더라도, 남은 치킨집을 더 추가해서 `M`개로 만들어도 도시의 치킨 거리는 나빠지지 않는다.

결론적으로 답은 항상 정확히 `M`개를 선택한 경우 안에 존재한다.

## 현재 종료 조건에 대한 피드백

현재 코드는 다음 조건에서 답을 갱신한다.

```python
if chick_idx == len(chicks) - 1 or selected_count == m:
```

`selected_count == m`일 때 종료하는 것은 좋다. 이미 `M`개를 골랐으므로 더 탐색할 필요가 없다.

다만 `chick_idx == len(chicks) - 1`일 때도 답을 갱신하면, `M`개보다 적게 선택한 상태도 답 후보에 들어간다. 위에서 설명한 이유 때문에 최적해는 정확히 `M`개 선택 상태 중에 존재하므로, 답 갱신은 `selected_count == m`일 때만 해도 충분하다.

또한 남은 치킨집을 모두 골라도 `M`개를 채울 수 없는 경우는 더 탐색할 필요가 없다.

```text
selected_count + 남은 치킨집 수 < M
```

이 조건을 만족하면 가지치기할 수 있다.

## BFS로도 풀 수 있나

상태를 큐에 넣고 탐색하면 BFS처럼 풀 수도 있다.

```text
현재 확인할 치킨집 인덱스
선택한 치킨집 개수
현재까지의 집별 최소 치킨거리
```

이 상태를 큐에 넣으면 `0`개 선택 상태, `1`개 선택 상태, ..., `M`개 선택 상태를 너비 우선으로 탐색할 수 있다.

다만 이 문제는 최단거리 문제가 아니다. BFS의 장점인 "먼저 도달한 상태가 최적"이라는 성질을 사용할 수 없다. 결국 `M`개를 고르는 모든 후보를 비교해야 하므로, 일반적으로는 DFS 백트래킹이나 `combinations`가 더 자연스럽다.

## 개선하면 좋은 점

현재 풀이는 정답성 측면에서는 좋아 보이고, 예제와 작은 랜덤 케이스도 통과했다. 다만 다음 부분을 다듬으면 코드 의도가 더 분명해진다.

- `answer = 0` 대신 `float("inf")`를 사용하면 최솟값 갱신 의도가 더 명확하다.
- `chick_idx`를 "이미 반영한 마지막 치킨집"으로 쓰기보다, `next_idx`처럼 "다음에 볼 치킨집"으로 잡으면 초기 stack 특수 처리를 줄일 수 있다.
- 답 갱신은 `selected_count == m`일 때만 하는 구조가 더 명확하다.
- 남은 치킨집 수로 `M`개를 채울 수 없는 상태는 가지치기할 수 있다.
- 단순성과 가독성을 우선한다면 `combinations` 풀이도 이 문제 제한에서는 충분히 좋은 선택이다.

## 정리

현재 백트래킹 풀이는 "집별 최소 치킨거리 상태를 재활용한다"는 점에서 좋은 시도다. 단순 `combinations`보다 계산을 재활용하려는 의도가 분명하다.

다만 이 문제는 치킨집 수와 집 수 제한이 작아서 `combinations` 풀이도 충분히 빠르다. 따라서 실전에서는 코드 단순성을 우선해 `combinations`를 선택할 수도 있고, 백트래킹 연습 관점에서는 현재 방식처럼 상태를 들고 가며 갱신하는 풀이도 의미가 있다.
