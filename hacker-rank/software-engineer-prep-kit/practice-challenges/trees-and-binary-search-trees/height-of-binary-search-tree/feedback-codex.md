# Height of Binary Search Tree 피드백

## 문제 조건 요약

- PDF는 텍스트 추출이 되지 않아 이미지 렌더링으로 조건을 확인했다.
- `values[i]`는 `i`번 노드의 값이고, `leftChild[i]`, `rightChild[i]`는 각각 `i`번 노드의 왼쪽/오른쪽 자식 인덱스다.
- `-1`은 해당 방향의 자식이 없다는 뜻이다.
- 높이는 루트에서 리프까지 지나가는 노드 수의 최댓값이다. 이 문제에서는 간선 수가 아니라 노드 수를 센다.

## `leftChild`, `rightChild` 해석

예제의 배열이 다음과 같다면:

```text
values     = [4, 2, 6, 1, 3, 5, 7]
leftChild  = [1, 3, 5, -1, -1, -1, -1]
rightChild = [2, 4, 6, -1, -1, -1, -1]
```

`leftChild[0] = 1`은 "값이 1인 노드가 왼쪽 자식"이라는 뜻이 아니다. **0번 노드의 왼쪽 자식이 1번 인덱스 노드**라는 뜻이다.

트리로 그리면 다음과 같다.

```text
        4        <- index 0
      /   \
     2     6     <- index 1, 2
    / \   / \
   1   3 5   7   <- index 3, 4, 5, 6
```

Sample Input 1은 다음 배열로 해석된다.

```text
values     = [5, 3]
leftChild  = [1, -1]
rightChild = [-1, -1]
```

트리 구조는 다음과 같다.

```text
    5        <- index 0
   /
  3          <- index 1
```

## 첫 번째 풀이 피드백

처음 풀이에서는 `go_left`, `go_right`를 따로 두고 방향별로 내려가려고 했다.

이 접근에서 가장 큰 문제는 자식 인덱스가 `-1`일 때도 반대쪽 탐색 과정에서 `leftChild[-1]` 또는 `rightChild[-1]`처럼 접근할 수 있다는 점이다. Python에서 `arr[-1]`은 에러가 아니라 마지막 원소를 의미하므로, 없는 자식을 잘못된 노드로 해석할 위험이 있다.

또 한 가지 문제는 한쪽 자식이 없는 노드에서 다른 쪽 자식을 계속 확인해야 하는데, 방향별 함수가 이 흐름을 복잡하게 만든다는 점이었다.

직접 확인한 반례:

```text
values     = [0, 1, 2]
leftChild  = [1, -1, -1]
rightChild = [-1, 2, -1]
```

트리:

```text
0
/
1
 \
  2
```

정답 높이는 `3`인데, 처음 풀이에서는 중간 노드의 오른쪽 자식만 있는 경우를 놓쳐 `2`가 나왔다.

## 두 번째 풀이 피드백

`go_down(idx, cur_depth)` 하나로 합친 방향은 좋았다. 현재 노드에서 왼쪽과 오른쪽 자식을 모두 확인하는 구조가 트리 높이 계산에 더 자연스럽기 때문이다.

다만 재귀 풀이에는 Python 재귀 깊이 제한 문제가 남았다.

직접 확인한 경계 입력:

```text
n = 1000
모든 노드가 왼쪽 자식으로만 이어진 트리
```

이 경우 논리적인 정답은 `1000`이지만, 재귀 호출이 깊어져 `RecursionError: maximum recursion depth exceeded`가 발생했다.

## 최종 스택 풀이 리뷰

최종 풀이에서는 재귀 대신 `stack`을 사용했다.

```python
stack = [(0, 1)]
answer = 1

while stack:
    i, depth = stack.pop()
    answer = max(answer, depth)

    if leftChild[i] != -1:
        stack.append((leftChild[i], depth + 1))
    if rightChild[i] != -1:
        stack.append((rightChild[i], depth + 1))
```

이 방식은 현재 문제에 잘 맞는다.

- `stack`에는 앞으로 방문할 노드 인덱스와 현재 깊이를 함께 넣는다.
- 자식이 있으면 깊이를 `1` 증가시켜 스택에 넣는다.
- 모든 노드를 방문하면서 가장 큰 깊이를 `answer`에 저장한다.
- 재귀를 쓰지 않으므로 한쪽으로 길게 치우친 트리에서도 `RecursionError`가 나지 않는다.

`values`를 높이 계산에 사용하지 않는 것도 맞다. 이 문제에서는 이미 `leftChild`, `rightChild`가 트리 구조를 직접 알려주고 있기 때문에, 노드 값으로 BST를 다시 만들 필요가 없다.

## 직접 확인한 테스트

다음 케이스들을 직접 실행해 확인했다.

```text
빈 트리             -> 0
노드 1개            -> 1
Sample Input 1      -> 2
균형 잡힌 7개 트리   -> 3
오른쪽 자식만 있는 반례 -> 3
왼쪽 체인 1000개     -> 1000
오른쪽 체인 1000개   -> 1000
무작위 트리 20개      -> 기준 풀이와 일치
```

## 복잡도

- 시간 복잡도: `O(n)`
- 공간 복잡도: `O(n)`

각 노드를 최대 한 번씩 방문한다. 스택에는 아직 방문하지 않은 노드들이 들어가므로 추가 공간은 최악 `O(n)`이다.

## 다음 문제에 적용할 점

트리 문제에서는 값보다 먼저 **연결 구조**를 봐야 한다. 특히 `leftChild[i]`, `rightChild[i]`처럼 인덱스 기반으로 트리가 주어진 경우에는 값을 비교해서 트리를 다시 만들지 말고, 주어진 연결 정보를 그대로 따라가면 된다.

또 Python에서는 트리 깊이가 커질 수 있으면 재귀 풀이가 맞는 아이디어여도 런타임에서 막힐 수 있다. 제한이 `1000` 근처까지 열려 있고 한쪽으로 치우친 트리가 가능하다면, 반복문과 스택을 쓰는 풀이가 더 안전하다.
