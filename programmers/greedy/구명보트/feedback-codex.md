# 구명보트 피드백

## 문제 조건 요약

- 구명보트는 한 번에 최대 2명까지 탈 수 있다.
- 보트에 탄 사람들의 몸무게 합은 `limit` 이하여야 한다.
- 사람 수는 최대 50,000명이므로, 모든 사람을 여러 번 훑는 방식은 시간 초과가 날 수 있다.

## 이전 풀이의 성능 문제

이전 풀이에서는 사람을 구조했는지 `rescued` 배열로 표시하고, 보트 하나를 만들 때마다 전체 사람 목록을 처음부터 다시 훑었다.

```python
while rescued_cnt < n:
    for i in range(n):
        ...
```

이 구조는 최악의 경우 보트 수만큼 `n`명을 반복해서 확인하므로 `O(n^2)`에 가까워진다. 사람 수가 최대 50,000명이기 때문에 효율성 테스트에서 시간 초과가 날 가능성이 크다.

또한 문제 조건상 보트에는 최대 2명만 탈 수 있으므로, 한 번에 여러 명을 채우는 방식은 정답성도 깨질 수 있었다.

## 현재 수정 방향

현재 코드는 사람을 오름차순으로 정렬한 뒤, 가장 가벼운 사람과 가장 무거운 사람을 양끝 포인터로 확인한다.

```python
people.sort()

s = 0
e = len(people) - 1
```

이 접근은 이 문제에 잘 맞는다.

- 가장 무거운 사람은 현재 남은 사람 중 반드시 이번 보트에 태워야 한다.
- 가장 가벼운 사람과 함께 탈 수 있으면 둘을 같이 보내는 것이 이득이다.
- 가장 가벼운 사람과도 함께 탈 수 없다면, 그 무거운 사람은 누구와도 함께 탈 수 없으므로 혼자 보내야 한다.

따라서 각 반복에서 가장 무거운 사람 `e`는 항상 처리되고, 같이 탈 수 있을 때만 가장 가벼운 사람 `s`도 함께 처리하면 된다.

## 성능 개선 효과

현재 방식은 정렬에 `O(n log n)`, 포인터 이동에 `O(n)`이 걸린다.

이전처럼 보트마다 전체 배열을 다시 훑지 않으므로, 최대 입력에서도 훨씬 안정적이다.

직접 확인한 테스트에서는 다음 케이스들이 통과했다.

- 예제: `[70, 50, 80, 50]`, `limit = 100`
- 예제: `[70, 80, 50]`, `limit = 100`
- 반례: `[40, 40, 40]`, `limit = 120`
- 반례: `[40, 50, 60]`, `limit = 150`
- 작은 입력 전수 비교
- 50,000명 규모의 성능 테스트

## 남은 개선점

현재 풀이의 알고리즘 방향은 맞다. 다만 포인터를 이동하는 순서는 조심해야 한다.

가장 무거운 사람 `e`를 먼저 줄인 뒤 비교하면, 원래 가장 무거운 사람이 아니라 그 다음 사람과 가장 가벼운 사람을 비교하게 된다.

```python
e -= 1

if people[s] + people[e] <= limit:
    s += 1
```

이렇게 되면 `[70, 80, 50]`, `limit = 100` 같은 예제에서도 `80`을 태운 뒤 `50 + 70`을 비교하는 꼴이 되어 답을 잘못 줄일 수 있다.

따라서 가장 무거운 사람과 가장 가벼운 사람을 먼저 비교한 뒤, 가장 무거운 사람은 항상 처리해야 한다.

```python
if people[s] + people[e] <= limit:
    s += 1
e -= 1
answer += 1
```

이렇게 쓰면 “가벼운 사람도 같이 탈 수 있으면 `s`를 이동하고, 무거운 사람은 항상 처리한다”는 흐름이 정확하게 드러난다.

## 정리

이번 수정으로 핵심 성능 문제는 해결됐다.

이 문제의 핵심은 “가장 무거운 사람을 기준으로, 가장 가벼운 사람과 같이 태울 수 있는지 확인한다”는 점이다. 이 방식은 보트당 최대 2명 조건과 잘 맞고, 정렬 후 한 번의 포인터 순회로 끝낼 수 있다.
