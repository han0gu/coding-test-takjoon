# 단어 변환 피드백

## 문제 조건 요약

- 한 번에 한 개의 알파벳만 바꿀 수 있다.
- 변환된 단어는 반드시 `words` 안에 있어야 한다.
- `begin`에서 `target`까지 가는 최소 변환 횟수를 반환한다.
- 변환할 수 없는 경우에는 `0`을 반환한다.

## 현재 풀이의 좋은 점

현재 풀이는 BFS를 사용하고 있어서 문제 성격과 잘 맞는다.

```python
q = deque([(begin, 0)])
```

이 문제는 한 번 변환할 때마다 비용이 항상 `1`이다. 따라서 BFS로 가까운 변환 단계부터 탐색하면, 어떤 단어에 처음 도달했을 때의 횟수가 그 단어까지의 최소 변환 횟수다.

또한 `visited`를 큐에 넣는 시점에 처리하고 있다.

```python
visited[i] = True
q.append((w, cnt + 1))
```

BFS에서는 이 방식이 좋다. 같은 단어가 큐에 여러 번 들어가는 것을 막을 수 있고, 처음 큐에 들어간 경로가 이미 최단 경로이기 때문이다.

`can_change()` 함수도 역할이 분명하다.

```python
def can_change(w1, w2):
    diff_cnt = 0
    for i in range(len(w1)):
        if w1[i] != w2[i]:
            diff_cnt += 1
            if diff_cnt > 1:
                return False
    return diff_cnt == 1
```

두 단어가 정확히 한 글자만 다른지 확인하는 함수라서 문제 조건을 잘 분리했다. 서로 다른 글자가 2개를 넘는 순간 바로 `False`를 반환하므로 불필요한 비교도 줄인다.

또한 `target`이 `words`에 없는 경우를 초반에 처리한다.

```python
if target not in words:
    return 0
```

문제 조건상 변환은 `words` 안에 있는 단어로만 가능하므로, 이 조건은 불가능한 경우를 빠르게 끝내는 좋은 처리다.

## 직접 확인한 테스트

다음 케이스들을 실행해서 확인했다.

```text
sample1: hit -> cog, words에 cog 있음 -> 4
sample2: hit -> cog, words에 cog 없음 -> 0
한 번에 변환되는 케이스 -> 1
여러 경로 중 최단 경로를 골라야 하는 케이스 -> 통과
target은 있지만 연결 경로가 없는 케이스 -> 0
```

현재 풀이의 정답성은 좋아 보인다.

## BFS에서는 처음 target을 만났을 때 바로 반환할 수 있다

현재 코드는 `target`을 만나는 순간 바로 반환한다.

```python
if cur == target:
    return cnt
```

하지만 BFS에서는 큐에서 꺼낸 순서가 변환 횟수가 적은 순서다. 따라서 `cur == target`이 처음 성립하는 순간의 `cnt`가 최소 변환 횟수다.

현재 구현처럼 바로 반환하면 `answer` 변수가 필요 없어지고, target에 도달한 뒤 불필요한 탐색도 하지 않는다.

## target이 words에 없는 경우

문제 조건상 변환은 `words` 안에 있는 단어로만 가능하다. 따라서 `target`이 `words`에 없다면 애초에 도달할 수 없다.

초반에 다음 조건을 넣을 수 있다.

```python
if target not in words:
    return 0
```

현재 구현은 이 조건을 이미 반영하고 있어서 불가능한 경우를 빠르게 끝낼 수 있고, 문제 조건도 코드에 명확하게 드러난다.

## can_change()에서 할 수 있는 작은 최적화

현재 `can_change()`는 서로 다른 글자가 2개를 넘으면 바로 종료한다. 단어 길이가 최대 `10`이라 끝까지 비교해도 충분하지만, 이 정도의 조기 종료는 가독성을 크게 해치지 않으면서 조건을 더 직접적으로 보여준다.

핵심은 서로 다른 글자가 2개 이상이면 이미 변환 불가능하다는 점이다.

```python
if diff_cnt > 1:
    return False
```

이 최적화는 필수는 아니지만, 현재 코드처럼 짧게 들어가면 괜찮다.

## 정리

현재 풀이는 BFS 방향이 맞고, 방문 처리 타이밍도 좋다. 정답성 측면에서도 샘플과 추가 테스트를 통과했다.

다음 문제에 가져가면 좋은 핵심은 두 가지다.

- BFS 특성을 이용해 `target`을 처음 만났을 때 바로 `return cnt` 하기.
- 불가능한 조건은 초반에 처리해 탐색 범위를 줄이기.

현재 풀이에는 이 두 가지가 잘 반영되어 있다.
