# 여행 경로 피드백

## 문제 조건 요약

- 항상 `ICN` 공항에서 출발한다.
- 주어진 항공권은 모두 정확히 한 번씩 사용해야 한다.
- 가능한 경로가 여러 개라면 전체 경로를 사전순으로 비교했을 때 가장 앞서는 경로를 반환한다.

## 이전 BFS/상태복사 풀이의 좋은 점

이전 풀이에서는 공항 방문 여부가 아니라 티켓 사용 여부를 관리했다.

```python
visited_idx
```

이 방향은 맞다. 이 문제에서는 같은 공항을 여러 번 방문할 수 있으므로, 공항 기준 `visited`를 쓰면 안 된다. 중요한 것은 "어떤 티켓을 이미 사용했는가"다.

예를 들어 다음과 같은 경로도 가능하다.

```text
ICN -> ATL -> ICN -> SFO
```

따라서 방문 상태는 공항이 아니라 티켓 단위로 관리해야 한다.

또한 이전 BFS 풀이에서는 각 큐 상태마다 `visited_idx`와 `cur_answer`를 복사해서 들고 갔다.

```python
new_visited_idx = [*visited_idx]
new_answer = [*cur_answer]
```

이 방식은 명시적인 rollback 코드는 없지만, 상태를 경로별로 분리해서 들고 가는 방식이다. 따라서 개념적으로는 백트래킹과 같은 "경로별 티켓 사용 상태 관리"를 하고 있다고 볼 수 있다.

## 왜 일반 visited로는 부족한가

이 문제는 일반적인 최단거리 BFS 문제가 아니다. 목표는 가장 짧은 경로를 찾는 것이 아니라, 모든 티켓을 정확히 한 번씩 사용하는 경로를 구성하는 것이다.

같은 공항에 도착했더라도 이미 사용한 티켓 목록이 다르면 이후 가능한 경로가 달라진다.

```text
현재 공항만 같음
하지만 사용한 티켓 목록이 다름
=> 서로 다른 상태
```

그래서 단순히 전역 `visited` 하나를 영구적으로 유지하면 안 된다. 어떤 경로에서 사용한 티켓이 실패 경로의 일부였다면, 다른 경로에서는 그 티켓을 다시 사용할 수 있어야 한다.

구현 방식은 크게 두 가지가 가능하다.

- DFS 백트래킹: 티켓을 사용 표시했다가 실패하면 되돌린다.
- 현재 풀이처럼 상태 복사: 각 경로마다 사용한 티켓 목록을 따로 들고 간다.

이전 BFS/상태복사 풀이는 두 번째 방식이다.

## 사전순 정렬 기준

경로 후보가 여러 개라면 전체 경로 리스트를 사전순으로 비교해야 한다.

이전에 다음처럼 정렬하면 문제가 생길 수 있다.

```python
answer.sort(key=lambda x: (x[0], x[1]))
```

이 방식은 첫 번째, 두 번째 공항까지만 비교한다. 하지만 실제 경로는 그 뒤 공항까지 모두 비교해야 한다.

예를 들어 두 경로가 다음처럼 앞부분이 같을 수 있다.

```python
["ICN", "A", "B", "A", "C", "A"]
["ICN", "A", "C", "A", "B", "A"]
```

두 경로는 첫 번째, 두 번째 원소가 모두 같다. 따라서 세 번째 원소 `"B"`와 `"C"`까지 비교해야 올바른 사전순 판단이 가능하다.

전체 경로를 모아서 비교한다면 다음과 같이 처리해야 리스트를 앞에서부터 차례대로 비교한다.

```python
answer.sort()
return answer[0]
```

또는 다음처럼 쓸 수도 있다.

```python
return min(answer)
```

`min(answer)`는 숫자가 아니어도 동작한다. 리스트끼리 비교할 때는 `sort()`와 마찬가지로 앞 원소부터 차례대로 비교해서 가장 작은 리스트를 반환한다.

## 직접 확인한 테스트

최종 코드는 다음 케이스들을 통과했다.

```text
예제 1
예제 2
중복 티켓 케이스
전체 경로 사전순 비교가 필요한 케이스
```

특히 다음 케이스에서 올바르게 전체 경로를 비교한다.

```python
tickets = [
    ["ICN", "A"],
    ["A", "C"],
    ["A", "B"],
    ["B", "A"],
    ["C", "A"],
]
```

정답은 다음 경로다.

```python
["ICN", "A", "B", "A", "C", "A"]
```

현재 적용한 그래프 + 반복 스택 방식도 이 케이스를 올바르게 처리한다.

## PR 리뷰 분석: 재귀 DFS의 한계

PR 리뷰에서 Gemini는 이전 DFS 백트래킹 풀이가 항공권 수 최대 10,000개 조건에서 안전하지 않을 수 있다고 지적했다. 이 지적은 타당하다.

### 1. `RecursionError`는 실제로 재현된다

이전 풀이의 `dfs()`는 티켓을 하나 사용할 때마다 재귀 깊이가 1씩 늘어난다.

```python
if dfs(end):
    return True
```

파이썬의 기본 재귀 깊이 제한은 보통 약 1,000이므로, 다음처럼 한 줄로 이어지는 유효한 경로만 있어도 항공권이 1,100장 정도가 되면 `RecursionError`가 발생할 수 있다.

```text
ICN -> AAA -> AAB -> AAC -> ...
```

문제 제한은 항공권 수가 최대 10,000개이므로, 예제와 작은 추가 테스트를 통과하더라도 제한 최댓값 기준으로는 위험하다.

`sys.setrecursionlimit()`을 높이면 재귀 깊이 문제는 어느 정도 피할 수 있다. 하지만 재귀 호출 구조 자체와 아래의 시간 복잡도 문제까지 함께 해결하는 방법은 아니다.

### 2. 시간 복잡도도 넉넉하지 않다

이전 DFS는 매 단계마다 전체 `tickets` 리스트를 앞에서부터 훑는다.

```python
for i, (start, end) in enumerate(tickets):
```

따라서 경로가 거의 한 가지로 정해져 있어도, 티켓 수가 `N`개라면 대략 `N`번의 깊이에서 매번 `N`개 티켓을 확인하게 되어 `O(N^2)`에 가까운 스캔이 생길 수 있다.

또한 잘못된 후보를 먼저 선택했다가 되돌아오는 경우가 많으면 백트래킹 탐색량은 더 커질 수 있다. 그래서 이 풀이는 "티켓 단위 visited와 rollback을 이해하기 좋은 풀이"이지, 최대 10,000장 조건까지 가장 안정적인 구조라고 보기는 어렵다.

### 3. 더 안정적인 방향: 그래프와 반복 스택

이 문제는 항공권을 방향 간선으로 보고, 모든 간선을 정확히 한 번씩 사용하는 경로를 만드는 문제로 볼 수 있다. 이런 관점에서는 오일러 경로를 만드는 Hierholzer 알고리즘을 사용할 수 있다.

핵심은 다음과 같다.

- `출발 공항 -> 도착 공항 목록` 형태의 그래프를 만든다.
- 도착지를 역순으로 넣어두고 `pop()`하면 사전순으로 가장 앞서는 공항을 먼저 꺼낼 수 있다.
- 재귀 대신 `stack`을 사용하므로 긴 경로에서도 재귀 깊이 제한에 걸리지 않는다.
- 모든 티켓은 그래프의 간선으로 한 번씩만 빠져나오므로, 정렬 이후 간선 처리 자체는 `O(N)`이다.

최종 반영한 개선 구조는 다음과 같다.

```python
from collections import defaultdict


def solution(tickets):
    graph = defaultdict(list)

    for start, end in sorted(tickets, reverse=True):
        graph[start].append(end)

    stack = ["ICN"]
    route = []

    while stack:
        cur = stack[-1]

        if graph[cur]:
            stack.append(graph[cur].pop())
        else:
            route.append(stack.pop())

    return route[::-1]
```

이 방식은 정렬에 `O(N log N)`, 그래프 순회에 `O(N)`이 걸린다. 공간은 그래프와 스택, 경로를 저장하므로 `O(N)`이다.

다만 이 방식은 "오일러 경로"라는 관점이 필요해서 처음에는 DFS 백트래킹보다 덜 직관적일 수 있다. 학습 단계에서는 먼저 티켓 사용 여부와 rollback을 이해하고, 그다음 제한 최댓값까지 고려하는 최적화 풀이로 넘어가면 좋다.

## 상태복사 구조의 아쉬운 점

상태복사 풀이는 가능한 완성 경로를 모두 `answer`에 모은 뒤 정렬한다.

```python
answer.append(cur_answer)
answer.sort()
return answer[0]
```

이 방식은 정답성 측면에서는 괜찮지만, 가능한 경로 수가 많아지면 메모리를 많이 사용할 수 있다. 모든 완성 경로를 저장해야 하기 때문이다.

상태복사 풀이에서 한 단계 개선하면, 티켓을 사전순으로 정렬해둔 뒤 DFS 백트래킹으로 탐색하고 모든 티켓을 사용한 첫 번째 경로를 바로 반환하는 방식이 더 자연스럽다.

```text
티켓을 사전순으로 정렬
ICN에서 DFS 시작
사용 가능한 티켓을 사전순으로 시도
모든 티켓을 사용한 첫 번째 경로 반환
```

이렇게 하면 모든 정답 후보를 저장하고 정렬할 필요가 없다.

## 상태복사 풀이의 변수명 피드백

상태복사 풀이에서 두 번째 값은 `idx`라는 이름을 쓰고 있었다.

```python
q = deque([('ICN', 0, [], ['ICN'])])
```

하지만 이 값은 실제로 "검토할 티켓 인덱스"가 아니라 "지금까지 사용한 티켓 개수"에 가깝다.

따라서 다음과 같은 이름이 더 정확하다.

```python
used_count
```

`visited_idx`도 리스트라서 다음 검사는 선형 탐색이다.

```python
if i not in visited_idx:
```

`visited_idx`의 길이가 길어질수록 앞에서부터 하나씩 확인해야 한다. 티켓 인덱스로 바로 사용 여부를 확인하려면 boolean 리스트가 더 적합하다.

```python
visited = [False] * len(tickets)

if not visited[i]:
    ...
```

리스트 인덱스 접근은 `O(1)`이고, `i not in visited_idx`는 `O(k)`다. 다만 현재처럼 상태를 복사해서 큐에 넣는 방식에서는 boolean 리스트도 매번 복사해야 하므로, 구조 자체는 여전히 무거울 수 있다.

## 중간 단계: DFS 백트래킹으로 개선한 구조

중간 단계로는 티켓을 먼저 사전순으로 정렬한 뒤, DFS 백트래킹으로 첫 완성 경로를 반환하는 구조가 자연스럽다.

```python
tickets.sort()
visited = [False] * len(tickets)
route = ["ICN"]
```

정렬된 티켓을 앞에서부터 확인하므로, 같은 출발 공항에서 갈 수 있는 후보를 사전순으로 시도하게 된다. 이때 모든 티켓을 사용한 첫 번째 경로가 곧 사전순으로 가장 앞서는 경로다.

백트래킹에서는 하나의 `visited`와 하나의 `route`를 공유하면서, 선택한 티켓이 실패 경로로 이어지면 상태를 되돌린다.

```python
visited[i] = True
route.append(end)

if dfs(end):
    return True

route.pop()
visited[i] = False
```

이 방식은 이전 상태복사 풀이와 비교했을 때 다음 장점이 있다.

- 모든 완성 경로를 `answer`에 저장하지 않는다.
- `min(answer)`로 다시 비교하지 않아도 된다.
- `visited`와 `route`를 매번 복사하지 않는다.
- 사전순으로 탐색하므로 첫 완성 경로를 바로 반환할 수 있다.

하지만 재귀 깊이와 최대 입력 크기까지 고려하면, 최종 코드처럼 반복 스택을 사용하는 그래프 풀이가 더 안전하다.

## 정리

핵심은 공항 방문 여부가 아니라 티켓 사용 여부를 관리해야 한다는 점이다. 이전 상태복사 풀이도 정답성은 좋았지만, 가능한 모든 완성 경로를 저장한 뒤 비교해야 했다.

DFS 백트래킹 구조에서는 다음 흐름이 더 명확하다.

- 티켓을 사전순으로 정렬한다.
- 티켓 단위 `visited`를 사용한다.
- 경로가 실패하면 `visited`와 `route`를 되돌린다.
- 모든 티켓을 사용한 첫 번째 경로를 반환한다.

학습 단계의 전형적인 풀이는 "티켓 visited + DFS 백트래킹 + 사전순 첫 경로 반환"에 가깝다. 다만 항공권 수 최대 10,000개 조건까지 안정적으로 고려하면, 최종 코드처럼 "그래프 + 반복 스택 + Hierholzer 알고리즘" 방향이 더 안전하다.
