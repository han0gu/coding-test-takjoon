# 가장 큰 수 피드백

## 문제 조건 요약

- `0` 또는 양의 정수가 담긴 배열 `numbers`가 주어진다.
- 숫자들을 이어 붙여 만들 수 있는 가장 큰 수를 문자열로 반환해야 한다.
- `numbers`의 길이는 최대 `100,000`이므로 모든 순열을 직접 만들 수 없다.
- 정답이 매우 클 수 있으므로 숫자가 아니라 문자열로 다뤄야 한다.

## 처음 순열 풀이의 문제

처음 접근은 `permutations(numbers, len(numbers))`로 모든 순서를 만들어 보고, 그중 가장 큰 값을 고르는 방식이었다.

이 방식은 아이디어 자체는 직관적이지만, 시간 복잡도가 `N!`이 된다. 예제처럼 원소가 3개일 때는 가능하지만, 입력 길이가 조금만 커져도 계산할 수 없다.

또한 중간에 다음처럼 전체 문자열을 `int`로 바꾸는 것도 문제가 된다.

```python
answer = max(answer, int(s))
```

문제에서는 정답이 매우 클 수 있기 때문에 문자열로 반환하라고 되어 있다. Python 3.11 기준으로는 너무 긴 숫자 문자열을 `int()`로 변환할 때 자리수 제한에 걸려 `ValueError`가 발생할 수 있다.

즉, 이 문제에서는 완성된 큰 수 전체를 정수로 바꾸어 비교하면 안 된다.

## 현재 풀이의 좋아진 점

현재 풀이는 순열을 전부 만들지 않고, 문자열 리스트를 정렬하는 방향으로 바뀌었다.

```python
ls = [str(x) for x in numbers]
ls.sort(key=cmp_to_key(compare), reverse=True)
```

이 방향은 맞다. 이 문제의 핵심은 모든 순열을 만드는 것이 아니라, 두 숫자 문자열의 상대적 순서를 정하는 것이다.

또한 모든 값이 `0`인 경우를 처리한 점도 좋다.

```python
return ''.join(ls) if ls[0] != '0' else '0'
```

예를 들어 `[0, 0, 0]`의 결과는 `"000"`이 아니라 `"0"`이어야 한다.

## 현재 compare 기준의 문제

현재 `compare()`는 길이가 다른 문자열을 마지막 문자로 채워서 길이를 맞춘 뒤 비교한다.

```python
elif l1 > l2:
    s2 += s2[-1] * (l1 - l2)
    return int(s1) - int(s2)
else:
    s1 += s1[-1] * (l2 - l1)
    return int(s1) - int(s2)
```

하지만 이 방식은 실제로 두 숫자를 이어 붙였을 때 어느 순서가 더 큰지와 항상 일치하지 않는다.

반례는 다음과 같다.

```python
numbers = [100, 10]
```

현재 풀이의 결과는 다음이다.

```text
10010
```

하지만 정답은 다음이다.

```text
10100
```

왜냐하면 두 숫자의 순서는 다음처럼 비교해야 하기 때문이다.

```text
"100" + "10" = "10010"
"10" + "100" = "10100"
```

`"10100"`이 더 크므로 `"10"`이 `"100"`보다 앞에 와야 한다.

## 올바른 비교 기준

두 문자열 `a`, `b`의 순서를 정할 때는 숫자 자체의 크기나 길이 보정값을 비교하면 안 된다.

대신 다음 두 값을 비교해야 한다.

```text
a + b
b + a
```

예를 들어 `a = "100"`, `b = "10"`이면:

```text
a + b = "10010"
b + a = "10100"
```

따라서 `b`가 `a`보다 앞에 와야 한다.

이 기준은 문제의 목표와 직접 연결된다. 결국 두 숫자의 상대적 순서는 "둘을 어떤 순서로 이어 붙였을 때 전체 문자열이 더 커지는가"로 판단해야 한다.

## int 변환을 피해야 하는 이유

문자열 비교만으로도 충분하다.

```text
"10100" > "10010"
```

두 문자열의 길이가 같다면 문자열 사전식 비교는 숫자 크기 비교와 같은 결과를 낸다. `a + b`와 `b + a`는 항상 길이가 같으므로, 굳이 `int()`로 바꿀 필요가 없다.

따라서 비교 함수 안에서도 `int()` 변환은 피하는 것이 좋다.

```text
좋은 방향: a + b 와 b + a를 문자열로 비교
피해야 할 방향: int(a + b), int(b + a)로 변환해서 비교
```

이렇게 하면 큰 정수 변환으로 인한 런타임 오류도 피할 수 있다.

## 정리

현재 풀이의 방향은 순열 완전탐색에서 정렬 기반 풀이로 잘 바뀌었다. 다만 비교 기준이 아직 정확하지 않다.

핵심은 다음이다.

- 모든 순열을 만들면 `N!`이라 시간 초과가 난다.
- 완성된 큰 수 전체를 `int`로 바꾸면 큰 입력에서 런타임 오류가 날 수 있다.
- 길이를 맞춰 패딩해서 비교하는 방식은 반례가 있다.
- 두 문자열 `a`, `b`는 `a + b`와 `b + a`를 비교해서 순서를 정해야 한다.

이 기준으로 `compare()`를 수정하면 현재 풀이 구조를 크게 바꾸지 않고 정답 방향으로 갈 수 있다.
