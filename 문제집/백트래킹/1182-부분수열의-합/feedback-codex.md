# 1182 부분수열의 합 피드백

## 문제 조건 요약

- `N`개의 정수로 이루어진 수열에서 크기가 양수인 부분수열을 고른다.
- 고른 원소들의 합이 `S`가 되는 경우의 수를 출력한다.
- `1 <= N <= 20`이므로 모든 원소에 대해 선택/비선택을 탐색하는 방식이 가능하다.

## 처음 풀이에서 놓쳤던 점

이 문제의 "부분수열"은 원소가 연속할 필요가 없다. 원래 수열에서 순서를 유지하면서 일부 원소를 고르면 된다.

예를 들어 수열이 다음과 같을 때:

```text
1 2 3
```

`1 3`도 부분수열이다. 두 원소가 원래 수열에서 붙어 있지는 않지만, 순서를 유지하면서 일부 원소를 고른 것이기 때문이다.

반대로 원소가 반드시 연속해야 하는 문제라면 보통 "연속 부분수열", "부분배열", "구간" 같은 표현이 나온다.

처음 풀이에서는 이전 후보 합을 재활용하면서 다음 원소를 붙이는 방식으로 접근했는데, 이 방식은 떨어져 있는 원소 조합을 충분히 만들지 못한다. 그래서 다음 입력에서 오답이 난다.

```text
3 4
1 2 3
```

정답은 `1`이다. `1 + 3 = 4`인 부분수열이 있기 때문이다.

## 현재 combinations 풀이

현재 풀이는 `itertools.combinations`를 사용해 크기 `1`부터 `N`까지의 모든 조합을 확인한다.

이 접근은 이 문제에서 정답 방향이다.

- 빈 부분수열을 제외한다.
- 연속하지 않은 원소 선택도 포함한다.
- `N <= 20`이라 전체 조합 수 `2^N - 1`을 확인할 수 있다.

실제로 다음 케이스들을 확인했을 때 현재 조합 풀이는 모두 맞는다.

```text
예제 입력 1 -> 1
1 2 3, target 4 -> 1
0 0 0, target 0 -> 7
원소 1개 match/no match
음수 target 케이스
```

다만 이 풀이는 백트래킹이라기보다는 `combinations`를 이용한 완전탐색에 가깝다. 조합 tuple을 만들고, 각 tuple마다 `sum()`을 다시 계산한다는 특징이 있다.

## 백트래킹으로 보면 어떻게 풀 수 있나

이 문제를 백트래킹으로 풀 때의 핵심은 각 원소마다 두 가지 선택지를 둔다는 것이다.

```text
현재 원소를 선택한다
현재 원소를 선택하지 않는다
```

그래서 탐색 상태는 보통 다음 세 가지로 표현할 수 있다.

```text
현재 보고 있는 인덱스
현재까지의 합
선택한 원소 개수
```

모든 원소를 다 본 시점에 다음 조건을 확인한다.

```text
현재 합 == S
선택한 원소 개수 > 0
```

두 조건을 모두 만족하면 정답을 1 증가시킨다. `S = 0`인 경우에도 아무것도 선택하지 않은 빈 부분수열은 세면 안 되기 때문에 `선택한 원소 개수 > 0` 조건이 필요하다.

## 예제 입력 1로 보는 백트래킹 흐름

예제 입력은 다음과 같다.

```text
N = 5, S = 0
nums = [-7, -3, -2, 5, 8]
```

정답이 되는 한 경로는 다음과 같다.

```text
시작: index = 0, sum = 0, count = 0

-7 선택 안 함
index = 1, sum = 0, count = 0

-3 선택함
index = 2, sum = -3, count = 1

-2 선택함
index = 3, sum = -5, count = 2

5 선택함
index = 4, sum = 0, count = 3

8 선택 안 함
index = 5, sum = 0, count = 3
```

이제 모든 원소를 다 봤고, `sum == 0`이며 `count > 0`이다. 따라서 이 경로는 정답에 포함된다.

이 경로가 의미하는 부분수열은 다음이다.

```text
[-3, -2, 5]
```

합은 `0`이다.

백트래킹은 이 경로 하나만 보는 것이 아니라, 각 원소에 대해 선택/비선택으로 갈라지는 모든 경로를 탐색한다.

## DFS와 백트래킹의 관계

DFS는 탐색 순서에 대한 개념이다.

```text
한 선택지를 깊게 끝까지 따라간 뒤, 이전 분기점으로 돌아와 다른 선택지를 본다.
```

백트래킹은 선택을 시도하고, 필요하면 이전 선택을 되돌린 뒤, 다른 선택을 시도하는 탐색 전략이다.

```text
선택한다
탐색한다
되돌아온다
다른 선택을 시도한다
```

그래서 백트래킹 문제는 대부분 DFS, 특히 재귀 DFS로 구현하는 경우가 많다. 하지만 DFS를 썼다고 해서 항상 백트래킹인 것은 아니다. 그래프 연결 요소 탐색처럼 단순히 방문만 하는 DFS는 보통 백트래킹이라고 부르지 않는다.

이 문제는 각 원소에 대해 선택/비선택을 시도하고, 한 선택 경로가 끝나면 이전 분기점으로 돌아와 다른 선택을 확인하므로 백트래킹으로 설명하기 좋다.

## rollback이 보이지 않을 수도 있는 이유

백트래킹이라고 해서 항상 `pop()` 같은 rollback 코드가 직접 보이는 것은 아니다.

상태를 값으로 넘기면 호출이 끝난 뒤 이전 상태가 그대로 유지된다.

```text
선택하는 호출: 다음 인덱스, 현재합 + nums[index], 선택 개수 + 1
선택하지 않는 호출: 다음 인덱스, 현재합, 선택 개수
```

이 방식에서는 `current_sum`이나 `count`를 직접 되돌리는 코드가 필요 없다. 함수 호출마다 별도의 값이 전달되기 때문이다.

반대로 `path` 같은 리스트를 공유하면서 실제로 선택한 원소 목록을 저장한다면 rollback이 명시적으로 필요하다.

```text
path에 원소 추가
탐색
path에서 원소 제거
```

즉, 백트래킹의 본질은 `pop()` 코드의 존재가 아니라 "선택을 시도한 뒤 이전 선택 지점으로 돌아와 다른 선택을 시도하는 구조"에 있다.

## combinations 풀이보다 백트래킹 풀이가 연습에 좋은 이유

현재 `combinations` 풀이는 이 문제 제한에서 충분히 좋은 풀이이다. 다만 백트래킹 유형을 익히는 관점에서는 재귀 DFS 풀이가 더 직접적이다.

- 문제의 선택/비선택 구조를 그대로 표현한다.
- 조합 tuple을 만들지 않아도 된다.
- 매번 `sum()`을 다시 계산하지 않고 누적합을 들고 내려갈 수 있다.
- 다른 백트래킹 문제로 확장하기 좋다.

따라서 이 문제에서는 `combinations`도 정답 풀이가 될 수 있지만, 백트래킹 연습을 위해서는 "각 원소를 선택하거나 선택하지 않는 결정 트리를 DFS로 탐색한다"는 방식으로 다시 구현해보는 것이 좋다.

## 시간 복잡도 관점

DFS 백트래킹으로 바꾼다고 해서 Big O 관점에서 완전히 다른 알고리즘이 되는 것은 아니다.

현재 `combinations` 풀이는 비어 있지 않은 모든 조합을 확인한다. 조합의 개수는 전체적으로 `2^N - 1`개다. 여기에 각 조합마다 `sum()`을 호출하므로, 최악의 경우 시간 복잡도는 대략 다음처럼 볼 수 있다.

```text
O(N * 2^N)
```

`sum(c)`가 조합의 길이만큼 다시 원소를 더하기 때문이다. 또한 `combinations`는 각 조합을 tuple 형태로 만들어주므로, 임시 tuple 생성 비용도 있다.

반면 DFS 백트래킹은 현재 합을 상태로 들고 내려갈 수 있다.

```text
선택함: current_sum + nums[index]
선택 안 함: current_sum
```

이렇게 하면 각 부분수열마다 `sum()`을 다시 계산하지 않아도 된다. 그래서 탐색해야 하는 선택/비선택 상태 수는 여전히 지수적이지만, 합 계산 자체는 누적해서 처리할 수 있다.

```text
O(2^N)
```

즉, 두 풀이 모두 모든 선택 경우를 확인한다는 점에서는 같은 완전탐색 계열이다. 백트래킹이 더 나은 점은 지수 탐색을 없애는 것이 아니라, 다음 비용을 줄이는 데 있다.

- 매번 `sum()`을 다시 계산하지 않는다.
- 조합 tuple을 계속 만들지 않아도 된다.
- 선택/비선택 구조가 문제 의미와 더 직접적으로 맞는다.

다만 이 문제는 음수도 포함될 수 있으므로 단순히 "현재 합이 이미 목표보다 크다" 같은 가지치기를 일반적으로 적용하기 어렵다. 따라서 백트래킹으로 풀어도 본질적으로는 모든 부분수열 후보를 확인하는 문제라고 보는 것이 맞다.
