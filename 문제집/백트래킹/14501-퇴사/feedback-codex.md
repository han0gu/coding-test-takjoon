# 14501 퇴사 피드백

## 문제 조건 요약

- `N`일 동안 상담 후보가 주어지고, 각 상담은 `T_i`일이 걸리며 수익은 `P_i`다.
- `i`일에 상담을 시작하면 `i + T_i - 1`일까지 다른 상담을 할 수 없다.
- 퇴사일 전까지 수행할 수 있는 상담 조합 중 최대 수익을 구한다.

## 현재 풀이의 좋은 점

현재 풀이는 상담을 선택하거나 선택하지 않는 두 경우를 모두 탐색한다. 이 문제를 백트래킹 관점으로 접근할 때 가장 중요한 방향은 맞다.

```text
현재 상담을 선택한다
현재 상담을 선택하지 않는다
```

또한 `stack`을 사용해서 재귀 대신 반복문으로 DFS를 구현한 점도 가능하다. `N <= 15`라서 모든 선택 경우를 탐색해도 충분히 감당할 수 있다.

## 구조가 복잡해진 이유

현재 풀이는 상담이 차지하는 날짜를 `occupied` 리스트에 저장하고, 새 상담을 선택할 때마다 겹치는 날짜가 있는지 확인한다.

```python
stack = [] # idx, acc, occupied
```

이 방식은 실제 달력처럼 날짜를 직접 점유 처리하므로 직관적인 면은 있다. 하지만 이 문제에서는 굳이 날짜별 점유 상태를 들고 다니지 않아도 된다.

상담은 항상 날짜 순서대로만 검토한다. 따라서 `idx`번째 상담을 선택했다면, 다음에 볼 수 있는 날짜는 바로 다음과 같다.

```text
idx + need
```

즉, 상담을 선택하는 순간 그 상담이 차지하는 기간 안의 날짜들은 자연스럽게 건너뛰면 된다.

## 더 단순한 상태 설계

이 문제에서 백트래킹 상태는 다음 두 가지만으로 충분하다.

```text
현재 검토할 날짜 인덱스
현재까지의 누적 수익
```

예를 들어 현재 `idx`일을 보고 있고, 상담 기간이 `need`, 수익이 `profit`이라면 선택지는 두 가지다.

```text
상담을 선택하지 않음:
    다음 상태 = idx + 1, 현재 수익 그대로

상담을 선택함:
    가능한 경우에만 다음 상태 = idx + need, 현재 수익 + profit
```

상담을 선택할 수 있는 조건도 단순하다.

```text
idx + need <= n
```

이 조건을 만족하면 퇴사 전까지 상담을 끝낼 수 있다. 만족하지 않으면 해당 상담은 선택하지 않고 넘어가면 된다.

## occupied가 필요 없는 이유

현재 `occupied` 방식은 매번 다음 작업을 한다.

- 선택한 상담의 날짜들을 리스트에 추가한다.
- 새 상담을 볼 때 해당 날짜들이 이미 점유되었는지 검사한다.
- 선택 분기마다 `occupied` 리스트를 복사한다.

하지만 날짜를 앞에서 뒤로만 진행한다면, 이미 지나간 날짜는 다시 선택할 일이 없다. 또한 상담을 선택한 경우에는 `idx + need`로 이동하므로, 겹치는 날짜를 애초에 후보로 보지 않게 된다.

그래서 날짜 점유 상태를 저장하는 대신, "다음에 검토할 인덱스"를 올바르게 이동시키는 것이 더 간단하다.

## 현재 구조의 비용

현재 방식도 `N <= 15`에서는 충분히 통과할 수 있다. 다만 구조적으로는 불필요한 비용이 있다.

- `occupied` 리스트를 계속 복사한다.
- `can_handle()`에서 날짜마다 `in occupied`를 검사한다.
- 초기 시작 지점을 별도로 만드는 코드가 필요해진다.
- 상담 선택 후 다음 날짜로 바로 이동하지 않기 때문에, 이미 점유된 날짜를 다시 후보로 보고 걸러내야 한다.

이 문제는 입력 크기가 작아서 성능보다 가독성 차이가 더 크다. 하지만 백트래킹 상태를 잘 잡으면 코드가 훨씬 짧아지고 실수할 지점도 줄어든다.

## 백트래킹 관점의 핵심

이 문제를 백트래킹으로 풀 때의 핵심은 "날짜를 점유했는가"가 아니라 "현재 날짜에서 어떤 선택을 할 것인가"다.

```text
현재 날짜의 상담을 안 한다 -> 다음 날짜로 이동
현재 날짜의 상담을 한다 -> 상담이 끝난 다음 날짜로 이동
```

이렇게 생각하면 별도의 `occupied` 배열이나 리스트 없이도 겹치는 상담을 자연스럽게 배제할 수 있다.

## 정리

현재 풀이는 정답성 측면에서는 괜찮지만, 상태를 실제 달력처럼 관리하면서 풀이가 필요 이상으로 복잡해졌다.

다음에 직접 고쳐볼 때는 `stack` 또는 재귀 DFS의 상태를 다음처럼 줄여보는 것이 좋다.

```text
idx: 현재 검토할 날짜
acc: 현재까지 얻은 수익
```

그리고 분기는 다음 두 가지로만 두면 된다.

```text
skip: idx + 1
take: idx + need
```

이 구조가 이 문제의 백트래킹 풀이를 가장 단순하게 만든다.
